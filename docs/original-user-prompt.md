# Original User Prompt

Нужно глубоко исследовать и спроектировать десктопное Windows-приложение для создания фоторобота человека по словесному описанию. Целевой сценарий: оператор работает со свидетелем, потерпевшим или заявителем и на основе описания внешности формирует ориентировочный портрет разыскиваемого или пропавшего человека. В приоритете первый рабочий этап: реалистичный MVP в виде приложения-конструктора, без попытки сразу делать полноценную генерацию лица нейросетью по свободному тексту.

Главное требование: не привязываться заранее к конкретному стеку, библиотекам или архитектурным фреймворкам. Сначала исследовать предметную область, похожие продукты, подходы, ограничения, а потом самостоятельно предложить наилучший путь реализации.

Что нужно исследовать

1. Изучить существующие системы фотороботов и близкие решения.


2. Посмотреть, как устроены классические facial composite systems и современные гибридные подходы.


3. Отдельно изучить такие ориентиры, как EvoFIT, EFIT-V и другие похожие продукты или исследования.


4. Понять, какие подходы лучше работают на практике:

сборка лица из компонентов;

создание стартового образа по описанию;

поэтапное уточнение оператором;

whole-face / iterative selection подходы.



5. Проверить, существуют ли:

открытые или доступные базы лиц, компонентов лица, шаблонов, фотороботов;

оффлайн-решения для перевода голоса в текст;

готовые наработки, которые можно использовать в рамках прототипа без облачной зависимости.



6. Сразу учитывать, что проект предполагается для чувствительной среды, поэтому нужно ориентироваться на полностью локальную работу без обязательной отправки данных наружу.



Что представляет собой проект Проект должен позволять оператору формировать фоторобот по описанию внешности человека. На первом этапе это не “нейросеть, которая рисует лицо по одному промпту”, а управляемый конструктор с удобной логикой работы. Пользователь должен иметь возможность:

ввести описание вручную;

в перспективе продиктовать описание голосом;

получить стартовый вариант лица;

пошагово отредактировать его;

сохранить итоговый фоторобот и промежуточные версии.


Цель MVP Сделать простой, понятный и демонстрабельный первый прототип, который уже решает практическую задачу: помогает составить фоторобот по словесному описанию без сложной генеративной модели.

Что должно быть в MVP

1. Десктопное приложение под Windows.


2. Экран создания нового фоторобота.


3. Форма ввода описания человека.


4. Нормализованный набор признаков внешности, с которыми работает система.


5. Базовый механизм подбора и сборки лица из шаблонных элементов.


6. Возможность ручной корректировки всех основных черт лица.


7. Возможность добавления особых примет.


8. Сохранение проекта, чтобы к нему можно было вернуться позже.


9. Экспорт итогового результата в удобный формат для демонстрации или печати.


10. Работа без обязательного интернета.



Какие признаки минимум нужно поддержать в MVP

пол;

примерный возраст;

форма лица;

линия подбородка;

форма и размер носа;

глаза;

брови;

губы;

уши;

волосы;

усы / борода;

цвет / тон внешности, если это релевантно выбранному стилю отображения;

шрамы, родинки, очки, другие заметные особенности.


Как должен работать MVP

1. Оператор создаёт новую карточку.


2. Вводит описание внешности в свободной форме и/или через набор полей.


3. Система преобразует описание в структурированные признаки.


4. На их основе собирается стартовый фоторобот.


5. Оператор вручную правит отдельные элементы до нужного сходства.


6. Итог сохраняется как версия фоторобота.



Что нужно продумать сразу, чтобы потом перейти ко второму этапу Нужно изначально заложить такую структуру проекта, чтобы после MVP можно было без полной переделки перейти к гибридной версии, где появятся:

голосовой ввод с оффлайн-транскрибацией;

более умная интерпретация свободного описания;

генерация нескольких стартовых вариантов;

итеративное уточнение лица через выбор наиболее похожих вариантов;
более “целостный” подбор лица, а не только редактирование по отдельным частям.


То есть MVP должен быть не тупиковым прототипом, а фундаментом для следующей стадии.

Второй этап, который нужно предусмотреть заранее После MVP система должна эволюционировать в гибридный вариант:

1. Ввод описания голосом и текстом.


2. Автоматическое извлечение признаков из свободной речи.


3. Выдача не одного, а нескольких стартовых вариантов лица.


4. Механизм уточнения: оператор или свидетель выбирает более похожие варианты, и система постепенно улучшает результат.


5. Сохранение истории итераций и сравнений.



Ограничения и принципы

Не делать ставку только на text-to-image подход.

Не завязывать базовую работоспособность на интернет.

Не проектировать систему как инструмент автоматической идентификации личности.

Позиционировать решение как инструмент помощи оператору при составлении ориентировочного фоторобота.

Учитывать, что данные могут быть чувствительными, поэтому локальность и контролируемость важнее “вау-эффекта”.


Что нужно получить от Codex на выходе

1. Краткий обзор предметной области и существующих решений.


2. Вывод, какой подход лучше всего подходит для первого рабочего MVP.


3. Формализацию функций MVP.


4. Предложение структуры сущностей и модулей проекта без жёсткой привязки к стеку.


5. План, как из MVP перейти к гибридной версии без переписывания всего с нуля.


6. Список рисков и слабых мест проекта.


7. Если возможно, подготовить уже начальный каркас проекта и первую версию интерфейса MVP.



Отдельное указание для исследования Нужно обязательно посмотреть в интернете существующие продукты, исследования и наработки по теме фотороботов и систем составления лица по описанию. В частности, изучить решения уровня EvoFIT, EFIT-V и другие аналогичные системы, а также проверить наличие:

оффлайн speech-to-text решений;

наборов лицевых компонентов;

открытых датасетов или материалов, полезных для создания прототипа;

подходов, которые реально подходят под локальное desktop-приложение.
