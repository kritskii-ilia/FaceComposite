# Архитектура и решения

Документ фиксирует, *почему* MVP сделан именно так, и как он переходит к этапу 2.
Дополняет исходный бриф [original-user-prompt.md](original-user-prompt.md).

## Контекст

- Ближайшая цель — демонстрация прототипа на выставке (РУДН).
- Долгосрочно — полноценное ПО для служб (РФ, затем шире).
- Среда чувствительная → приоритет: локальность и контролируемость, а не «вау».

## Решение 1. Стек: локальный web без сборки (с путём в нативность)

Рассматривались три варианта: остаться на WinUI 3 (исходный каркас), нативный
.NET/Avalonia, локальный web.

Выбран **локальный web** (HTML/SVG/JS, классические `<script>`, без бандлера):

- **Надёжность демо.** Открывается двойным кликом в Edge, который есть на любой
  Windows-машине. Нет тулчейна, который мог бы «не собраться» на чужом ноутбуке.
  (На целевой машине на момент старта не было ни .NET SDK, ни Node — любой нативный
  путь начинался бы с установки и рисков сборки WinUI из CLI без Visual Studio.)
- **Скорость до результата.** Рабочий вертикальный срез получен сразу, без
  инфраструктурной обвязки.
- **Не тупик.** Тот же фронтенд оборачивается в нативное окно Windows через
  **WebView2** или **Tauri** — это легитимная продакшен-архитектура. Вся логика
  (`traits/extract/render/store`) переиспользуется как есть.
- **Офлайн и приватность.** Нет сети, нет внешних зависимостей и CDN; данные
  только в браузерном хранилище / файлах пользователя.

Цена решения: иллюстративный (не фотореалистичный) стиль и пока не «настоящий»
нативный процесс. Оба пункта осознанно отнесены к этапу 2.

## Решение 2. Рендер — параметрический

Лицо НЕ собирается свопом готовых картинок. `render.js` рисует каждую черту кодом
из опорных точек, вычисленных от геометрии головы (`metrics`).

Почему не библиотека ассетов (DiceBear/Open Peeps и т.п.):

- Полноценная интеграция готового набора офлайн без сборщика — хрупкая.
- Свопу картинок чужд **плавный** контроль (бриф требует ручной корректировки всех
  черт) — а ползунки естественно ложатся на параметры генератора.
- Параметрический движок компактен, полностью офлайн и без лицензионных рисков.

Слои рисуются в фиксированном порядке (задние волосы → уши → голова → борода →
брови → глаза → нос → рот → очки → передние волосы → приметы) для корректных
перекрытий.

## Решение 3. Извлечение признаков — детерминированное

`extract.js` — словарь правил «регэксп → признак/значение» для русского языка,
плюс список *evidence* (что и по какому совпадению выставлено). Объяснимость важнее
«магии» на первом этапе (см. D-004 исходного каркаса).

Важный нюанс реализации: в JS `\w` **не** матчит кириллицу, поэтому правила
компилируются через `cyr()` с заменой `\w` на кириллический класс и с гибким
зазором перед ключевыми существительными («тёмные **короткие** волосы»).

## Модель данных (задел под этап 2)

`TraitProfile = { values, params, marks }`:

- `values` — категориальные выборы;
- `params` — непрерывные числовые корректировки (для ползунков);
- `marks` — особые приметы.

Состояние карточки целиком сериализуемо в JSON — это и формат проекта
(`.fcase.json`), и запись в localStorage. На этапе 2 рядом встаёт третий слой —
`confidence`/`evidence` на признак — не ломая `values`/`params`.

## Что переосмыслено относительно исходного каркаса

Исходный WinUI-каркас (от ChatGPT) был «архитектурой ради архитектуры»: 5 проектов
и DDD-слои при том, что проект **не собирался** (у `Rendering` не было ссылок на
зависимости, контракты рендера образовывали потенциальный цикл `Application ↔
Rendering`), а главная ценность — сам рендер — была заглушкой.

Здесь приоритеты развёрнуты: сначала работающий рендер + сквозной сценарий
(описание → портрет → правка → версия → экспорт), слои наращиваются по мере нужды.
